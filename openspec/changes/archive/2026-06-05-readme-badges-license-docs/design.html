技術決策 ／ 風險
技術決策
決策 1 ｜版本徽章選 GitHub release badge
使用 img.shields.io/github/v/release/…，在打 git tag 後自動同步版本號，不需要手動改 README。授權徽章用 img.shields.io/github/license/…，從 LICENSE 文件自動讀取授權類型。
✗ 棄用：靜態 badge → 每次版本升級要手動改 README，容易遺漏
✗ 棄用：npm badge → repo 尚未 publish，badge 顯示 404
決策 2 ｜圖表 table 只在 propose section 放一次
openspec-design-html 和 openspec-propose-design-mermaid-html 使用完全相同的 HTML 產生邏輯。重複放 table 代表日後新增圖表類型要改兩處，維護成本加倍。openspec-design-html section 改為一句引導 + 錨點連結。
風險
⚠ GitHub release badge 需要 git tag
若 repo 目前沒有任何 release tag，版本徽章顯示「No releases」而非版本號。
→ task 4.2 補充說明：實作後執行 git tag v1.2.0 && git push origin v1.2.0 建立第一個 tag
⚠ openspec-design-html 錨點連結依賴 section 標題文字
Markdown 錨點由標題文字自動產生。若未來 openspec-propose-design-mermaid-html 的 section 標題被改名，openspec-design-html 的錨點會失效。
→ 低機率，純文件連結，失效後只是連結失效，不影響功能
